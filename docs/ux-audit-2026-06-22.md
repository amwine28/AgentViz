## [1] ✅ FIXED · (CRITICAL) 2D graph draws ALL labels always-on with zero declutter — guaranteed unreadable smear at ~73 nodes
- **area:** 2D view  
- **file:** `/Users/aaronwinegrad/dev/AgentViz/ui/src/components/Graph.tsx:149-157, 187-188`

- **fix:** Port the 3D declutter approach into the tick handler. Minimally: only ever render labels for the selected node, nodes with a live operation badge, and the top-N by `activity` (e.g. N=12); hide the rest. Add a hover handler on each circle that toggles its own label visible on mouseenter/leave so any node is still inspectable.

## [2] ✅ FIXED · (HIGH) 2D node radius caps at 30px with no collision force — 73 nodes physically overlap, compounding label collisions
- **area:** 2D view  
- **file:** `/Users/aaronwinegrad/dev/AgentViz/ui/src/components/Graph.tsx:74-77, 133`  
- **evidence:** The simulation has only link/charge/center forces (lines 74-77) — there is no `d3.forceCollide`. This alone meaningfully reduces label overlap by spreading nodes.

## [3] ✅ FIXED · (HIGH) 2D Graph uses a generic force layout with no hierarchy/cluster force — a 1→72 star collapses into an unreadable hairball
- **area:** 2D view  
- **file:** `ui/src/components/Graph.tsx:75-77`

## [7] ✅ FIXED · (HIGH) 3D autoRotate keeps spinning during/after scroll-zoom because only the OrbitControls 'start' event stops it (wheel does not fire 'start')
- **area:** 3D view (Scene3D.tsx)  
- **file:** `/Users/aaronwinegrad/dev/AgentViz/ui/src/components/Scene3D.tsx:280-283`  
- **evidence:** Setup sets `controls.autoRotate = true; controls.autoRotateSpeed = 0.45;` then `controls.addEventListener?.('start', () => { if (!funRef.current) controls.autoRotate = false; })` (lines 280-282). In three.js OrbitControls the `'start'` event fires on pointer-down (rotate/pan drag), but mouse-WHEEL zoom dispatches `'start'`/`'end'` only in newer three versions and historically did not — and even when it does, the camera is simultaneously being orbited by autoRotate while the user tries to zoom, so the target keeps drifting. The user perceives the scene 'fighting' their input: they scroll to zoom but the field is still rotating. This is the most likely cause of the user feeling 3D controls are unreliable. There is also no `controls.update()` call in the animate loop tied to autoRotate damping, relying entirely on the library's internal loop.  
- **fix:** Make zoom (and any interaction) reliably halt autoRotate: in the animate loop or via a `'change'` listener, stop autoRotate on first user input — `controls.addEventListener?.('change', () => { if (!funRef.current && firstUserGesture) controls.autoRotate = false; })` is too noisy (autoRotate itself fires 'change'); instead add a `wheel` listener on the container: `el.addEventListener('wheel', () => { if(!funRef.current) controls.autoRotate = false; }, { passive: true })` alongside the existing 'start' handler. Best practice: stop autoRotate on ANY user gesture (pointerdown + wheel + touchstart) and optionally resume after N seconds of idle. Remember to remove the wheel listener in the cleanup return (line 367).
